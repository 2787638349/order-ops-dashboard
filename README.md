# 出行平台订单运营分析系统

## 项目简介

出行平台订单运营分析系统是一个基于出租车/出行平台订单数据构建的数据分析后台，面向订单运营、数据分析学习、课程设计展示和简历项目展示场景。系统以 NYC Yellow Taxi 行程数据为样例数据源，围绕订单规模、成交金额、客单价、行程效率、异常行程、支付方式、乘客数、距离区间、小时高峰和区域热点等指标进行统计分析与可视化展示。

本项目解决的问题是：将原始行程订单明细转化为可查询、可筛选、可展示的运营分析指标，帮助用户快速理解指定日期范围内的订单规模变化、收入表现、出行需求高峰、热门上下车区域和异常行程分布情况。

适合的业务场景包括：

- 出行平台订单运营看板
- 出租车行程数据分析后台
- 城市交通出行数据课程项目
- 数据可视化与后端聚合查询优化实践
- 求职简历中的数据分析后台项目展示

核心分析目标包括：

- 统计订单量、成交金额、平均客单价等核心经营指标
- 分析每日行程趋势和每小时出行高峰
- 观察支付方式、乘客数、距离区间等订单结构分布
- 识别异常行程并展示异常原因
- 分析上车区域和下车区域 Top10 热点
- 通过聚合统计表提升大数据量查询性能

## 功能模块

### 1. 登录认证

系统提供后台登录入口，前端登录成功后将 token 保存到本地，并通过 Axios 请求拦截器在后续接口请求中自动携带认证信息。当前演示账号配置在后端 `Config` 中。

### 2. 运营概览

运营概览页用于展示指定日期范围内的核心经营指标和趋势图表，包括：

- 总订单数
- 正常行程数
- 异常行程数
- 总成交金额
- 平均客单价
- 平均行程距离
- 平均行程时长
- 小费率

### 3. 日期范围筛选

系统支持按日期范围筛选运营指标，前端提供日期选择器和快捷筛选按钮，例如最近 3 天、最近 7 天、最近 14 天和整月数据。筛选条件会传递给后端分析接口，用于查询对应时间范围内的聚合统计结果。

### 4. KPI 指标卡片

首页通过 KPI 卡片集中展示订单规模、成交金额、平均客单价、行程效率和异常行程等核心指标，适合快速了解当前筛选条件下的运营状态。

### 5. 每日行程趋势

每日行程趋势图按日期统计行程数量和成交金额，帮助分析订单规模与收入随时间的变化情况。

### 6. 分布分析

分布分析模块基于聚合统计表展示多维度订单结构，包括支付方式、小时高峰、乘客数和距离区间等指标。

### 7. 支付方式占比

根据订单中的支付方式字段统计不同支付类型的行程数量，并通过饼图展示支付结构占比。

### 8. 每小时高峰分析

按上车时间的小时维度统计订单量，展示 0-23 点的行程分布，用于识别早晚高峰和出行活跃时段。

### 9. 乘客数分布

按乘客数量统计订单量，观察单人出行、多人同行等订单结构特征。

### 10. 距离区间分布

将行程距离划分为 `0-1`、`1-3`、`3-5`、`5-10`、`10-20`、`20+` 等区间，统计不同距离区间下的订单数量。

### 11. 区域热点分析

区域热点分析页展示上车区域 Top10 和下车区域 Top10，并结合区域名称、行政区和行程数量呈现热门出行区域。

### 12. 异常行程识别

系统在导入行程数据时会根据规则识别异常行程，并记录异常原因。当前异常规则包括：

- 总金额小于 0
- 车费金额小于 0
- 行程距离小于等于 0
- 乘客数量小于等于 0
- 上车时间晚于或等于下车时间
- 行程时长小于等于 0
- 行程时长超过 720 分钟
- 行程距离超过 100 英里

### 13. 行程记录管理

系统提供行程明细管理页面，支持行程列表分页查询、条件筛选、CSV 导入、CSV 导出和异常原因查看。筛选条件包括行程编号、支付方式、乘客数、上下车区域、日期范围、距离范围、金额范围和异常状态。

### 14. 聚合统计刷新

系统提供聚合统计刷新能力，可通过命令行脚本或前端按钮触发。刷新过程会基于 `taxi_trips` 明细表重新生成每日、小时、支付方式、乘客数、距离区间和区域热点等聚合统计表，用于提升看板查询性能。

## 数据库设计

本系统以行程订单表为核心事实表，记录每一笔出租车行程的时间、地点、乘客数、距离、金额、支付方式、行程时长和异常状态等明细信息。

当前项目中的核心物理表包括：

- `taxi_trips`：行程订单事实表，保存订单明细，是分析系统的数据核心。
- `taxi_zones`：地点区域维度表，保存 TLC LocationID 对应的行政区、区域名称和服务区域。
- `taxi_daily_stats`：每日聚合统计表，保存订单量、正常行程数、异常行程数、金额、距离、时长、客单价等指标。
- `taxi_hourly_stats`：小时分布聚合表，保存每日每小时行程数量。
- `taxi_payment_stats`：支付方式聚合表，保存不同支付方式的行程数量。
- `taxi_passenger_stats`：乘客数聚合表，保存不同乘客数量对应的行程数量。
- `taxi_distance_stats`：距离区间聚合表，保存不同行程距离区间的订单数量。
- `taxi_pickup_location_stats`：上车区域热点聚合表，保存上车地点维度的订单数量。
- `taxi_dropoff_location_stats`：下车区域热点聚合表，保存下车地点维度的订单数量。
- `taxi_stats_meta`：聚合统计任务状态表，保存统计刷新状态、最后刷新时间和提示信息。

从分析建模角度看，行程订单表是核心事实表；支付方式、地点区域、时间维度、供应商等信息是围绕订单展开的维度信息。当前代码中，地点区域已独立为 `taxi_zones` 维度表，支付方式、供应商和时间维度主要以内嵌字段或聚合统计口径体现。

聚合统计表用于提升查询性能。看板页面不直接对百万级明细数据进行复杂实时聚合，而是优先读取预先计算好的统计表，从而降低接口响应时间并提升前端图表加载稳定性。

## E-R 图说明

本系统以行程订单为核心实体，供应商、支付方式、地点区域、时间维度等实体围绕订单展开。

一个供应商可以对应多条订单；一种支付方式可以对应多条订单；一个地点区域可以作为多条订单的上车地点或下车地点；一个时间维度可以对应多条订单。

在当前物理实现中，`taxi_trips` 保存行程订单事实数据，`taxi_zones` 作为地点区域维度表分别关联订单的 `pickup_location_id` 和 `dropoff_location_id`。供应商编号、支付方式编号和时间字段存储在订单表中，并在聚合统计时转换为分析维度。

## 关系模型

供应商（供应商编号，供应商名称）

支付方式（支付方式编号，支付方式名称）

地点区域（地点编号，行政区，区域名称，服务区域）

时间维度（时间编号，日期，年份，月份，日期号，小时，星期）

行程订单（订单编号，供应商编号，支付方式编号，上车地点编号，下车地点编号，时间编号，上车时间，下车时间，乘客数量，行程距离，车费金额，小费金额，总金额，行程时长，是否异常）

说明：以上关系模型是面向课程设计和数据仓库建模表达的逻辑模型；当前代码中的物理实现以 `taxi_trips`、`taxi_zones` 和多张聚合统计表为主。

## 项目亮点

- 支持百万级出租车/出行订单明细数据处理。
- 使用 `taxi_trips` 作为核心事实表，围绕时间、支付方式、乘客数、距离和区域进行多维度分析。
- 使用聚合统计表提升查询性能，降低看板接口对明细表的实时聚合压力。
- 使用 ECharts 实现每日趋势、小时分布、支付方式占比、乘客数分布、距离区间分布和区域 Top10 等可视化分析。
- 支持日期范围筛选和快捷筛选，便于对比不同时间窗口下的运营表现。
- 支持异常行程识别，并记录可追踪的异常原因。
- 支持区域热点 Top10 分析，展示热门上车区域和下车区域。
- 前端 UI 采用数据分析后台风格设计，具备登录页、顶部导航、指标卡片、筛选面板、图表面板和明细表格。
- 提供 CSV 导入导出和 Parquet 批量导入脚本，兼顾演示数据和大规模原始数据导入。

## 运行步骤

### 1. 数据库准备

项目后端当前连接 MySQL，连接配置位于：

```text
backend/app/config.py
```

当前默认数据库连接为：

```text
mysql+pymysql://root:123456@127.0.0.1:3306/order_ops_db?charset=utf8mb4
```

运行前需要确保本地 MySQL 已启动，并已创建 `order_ops_db` 数据库。后端启动时会通过 `db.create_all()` 自动创建代码中定义的数据表。

### 2. 后端启动

进入后端目录：

```bash
cd backend
```

安装 Python 依赖：

```bash
pip install -r requirements.txt
```

启动 Flask 后端服务：

```bash
python run.py
```

后端默认运行地址：

```text
http://127.0.0.1:5000
```

### 3. 数据导入

导入地点区域维度数据：

```bash
python scripts/import_taxi_zones.py
```

导入 Parquet 行程数据：

```bash
python scripts/import_taxi_parquet.py data/yellow_tripdata_2025-01.parquet --batch-size 5000
```

生成或刷新聚合统计数据：

```bash
python rebuild_taxi_stats.py
```

按日期范围刷新聚合统计数据：

```bash
python rebuild_taxi_stats.py --start-date 2025-01-01 --end-date 2025-01-31
```

项目还提供了索引脚本：

```text
backend/scripts/add_taxi_trip_indexes.sql
```

该脚本用于为 `taxi_trips` 添加距离、时间、支付方式、乘客数和上下车区域相关索引。由于模型中已定义部分索引，是否手动执行该脚本需要结合当前数据库实际索引情况判断。

### 4. 前端启动

进入前端目录：

```bash
cd frontend
```

安装前端依赖：

```bash
npm install
```

启动开发服务：

```bash
npm run dev
```

构建生产环境静态资源：

```bash
npm run build
```

本地预览构建结果：

```bash
npm run preview
```

前端接口地址在 `frontend/src/api/request.js` 中配置，当前默认请求后端：

```text
http://127.0.0.1:5000
```

### 5. 登录说明

当前演示账号配置在 `backend/app/config.py` 中：

```text
用户名：admin
密码：123456
```

登录成功后前端会保存后端返回的 token，并在后续接口请求中通过 `Authorization: Bearer <token>` 发送给后端。

### 6. 环境变量配置说明

当前项目代码未读取 `.env` 文件，后端数据库连接、登录账号、token、端口等配置均写在 `backend/app/config.py` 的 `Config` 类中。

如需在不同环境部署，建议后续将以下配置迁移为环境变量：

- `SQLALCHEMY_DATABASE_URI`
- `FLASK_HOST`
- `FLASK_PORT`
- `FLASK_DEBUG`
- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`
- `ADMIN_TOKEN`
